# 需求设计文档

## 1. 需求名称
- 黑名单系统 V1

## 2. 背景
- 当前项目已具备账号、关系链、私聊、群聊与官方话题聊天室等基础社交能力。
- 按当前产品版本口径，**1.0 = 基础功能 + 拉黑系统**，因此黑名单能力是 1.0 的必备组成。
- 当前代码中仅存在黑名单占位测试，尚未形成真实用户能力。

## 3. 目标
- 允许用户将其他用户加入黑名单。
- 被拉黑后，双方之间的关键社交链路必须被阻断。
- 前端关系面板可查看、拉黑、取消拉黑。

## 4. 用户价值
- 防骚扰：避免继续收到不想建立联系的用户消息。
- 风险隔离：阻断私聊建立、继续私聊、拉群邀请。
- 自主控制：用户可自行管理黑名单并恢复关系。

## 5. 功能范围
- 包含：
  - 拉黑用户
  - 取消拉黑
  - 获取我拉黑的用户列表
  - 私聊创建拦截
  - 私聊消息发送拦截
  - 群聊创建时邀请拦截（创建者与被邀请人任一方向存在拉黑均禁止）
  - 关系页黑名单入口与基础提示
- 不包含：
  - 群内发言隔离
  - 历史消息撤回/隐藏
  - 被谁拉黑的可见性展示
  - 社群/频道级拉黑能力

## 6. 核心流程
1. 用户 A 在关系页对用户 B 执行“拉黑”。
2. 系统记录 A -> B 黑名单关系，并清理双方 follow 关系。
3. 若 A 尝试与 B 新建私聊，系统拒绝。
4. 若 A 与 B 已存在私聊，后续任一方在该私聊发消息时，系统拒绝。
5. 若 A 创建群聊并邀请 B，或 B 曾拉黑 A，系统拒绝。
6. 用户 A 可在黑名单页对 B 执行“取消拉黑”，之后恢复普通关系约束（但不会自动恢复关注）。

## 7. 交互/业务规则
- 黑名单为**单向记录**，但拦截策略按**双向生效**处理：
  - A 拉黑 B，A 与 B 任一方均不可建立私聊；
  - A 拉黑 B，A 与 B 任一方均不可在双方私聊内继续发消息；
  - A 拉黑 B，A 不能邀请 B 建群，B 也不能邀请 A 建群。
- 用户不能拉黑自己。
- 拉黑成功后，系统自动删除双方 follow 关系，避免继续出现在好友关系链中。
- 取消拉黑只移除黑名单，不自动恢复 follow/friend。
- 黑名单列表仅展示“我拉黑的人”，不展示“谁拉黑了我”。

## 8. 验收标准
- 可成功拉黑 / 取消拉黑 / 获取黑名单列表。
- 拉黑后私聊创建被拒绝。
- 拉黑后已有私聊内发送消息被拒绝。
- 拉黑后群聊创建时邀请对方被拒绝。
- 前端关系页可执行拉黑与取消拉黑，并能看到黑名单列表。
- 自动化测试覆盖服务层核心拦截逻辑。

## 9. 风险与待确认项
- 当前 V1 不处理“群内已存在共同成员时是否允许继续在群内交流”，先保持不拦截群消息。
- 当前 V1 不向用户暴露“是否被对方拉黑”的显式字段，仅通过失败提示体现。
