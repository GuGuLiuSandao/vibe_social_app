# 需求设计文档

## 1. 需求名称
- 社群管理 V1.1（自建群生命周期与身份权限）

## 2. 背景
- 当前项目已有账号体系、私聊/群聊、关注关系、官方固定话题聊天室等基础能力，但“群”仍更接近临时聊天会话，缺少面向运营和长期沉淀的社群管理能力。
- 产品版本已确认：**2.0 = 完整社群系统**。在 2.0 的建设中，1.1 版本优先补齐“自己建群后的管理能力”，先把自建群的生命周期、成员身份、基础资料与加入机制跑通。
- 当前用户只能“拉一个群聊”，不能真正创建、经营、扩展一个可持续存在的社群，无法满足玩家自组织、社群运营、官方/玩家双形态并存的目标。

## 3. 目标
- 支持用户创建“自建群”，并让群具备可持续存在的生命周期，而非一次性会话。
- 建立群内三层基础身份：群主、管理员、普通成员，并支持身份转换。
- 支持群资料管理：群头像、群简介、群公告。
- 支持群的加入与管理模式：私密群、申请加入群、公开群。
- 支持邀请、申请、审批等入群路径，形成完整的成员流转闭环。
- 为后续扩展更多社群类型打下统一模型基础，包括官方话题群、玩家自建群，以及后续新增类型。

## 4. 用户价值
- 对群主：可以真正“经营一个群”，而不只是临时拉人聊天。
- 对管理员：可以协助管理成员、处理申请、维护群秩序，降低群主管理负担。
- 对普通成员：能清晰理解群的加入方式、当前身份和可执行操作，减少使用门槛。
- 对平台：可以把官方群与玩家自建群纳入同一套社群框架，便于后续扩展频道、运营、审核、活动等能力。

## 5. 功能范围
- 包含：
  - 创建自建群
  - 群生命周期管理（创建、可运营、解散/关闭）
  - 群成员身份定义：群主 / 管理员 / 普通成员
  - 成员身份转换：任命管理员、取消管理员、群主转让
  - 成员退出、群主解散群、成员移除（踢出）
  - 群资料管理：群头像、群简介、群公告
  - 群类型定义与展示：官方话题群、玩家自建群，并预留扩展类型
  - 群加入模式：私密群、申请加入群、完全公开群
  - 群申请与审批流程
  - 群邀请流程与受邀加入
  - 基础操作权限控制与前端提示
- 不包含：
  - 群频道/子房间管理
  - 群禁言、封禁、黑名单、敏感词审核
  - 群推荐、搜索排序、发现页运营策略
  - 群等级、活跃值、勋章等成长体系
  - 复杂多级角色（如版主、巡查、机器人）
  - 官方运营后台

## 6. 核心流程
1. 用户创建一个自建群，填写群名称、头像、简介，并选择群加入模式。
2. 创建成功后，创建者自动成为群主，群进入“可运营”状态。
3. 群主可设置公告、任命管理员、邀请其他用户加入群。
4. 当群为“私密群”时，普通用户不能自由加入，只能通过邀请进入。
5. 当群为“申请加入群”时，用户可提交入群申请，群主或管理员审批后决定是否加入。
6. 当群为“公开群”时，用户可直接加入，无需审批。
7. 群主可将普通成员提升为管理员，也可撤销管理员身份。
8. 群主可将群主身份转让给当前群成员中的某一人，转让后原群主降为普通成员。
9. 成员可主动退群；管理员/群主可移除普通成员；只有群主可解散群。
10. 群解散后，群进入不可用状态，成员不可继续访问或发送消息。

## 7. 交互/业务规则

### 7.1 群生命周期
- 群至少包含以下状态：
  - `active`：正常可用
  - `closed` / `dissolved`：已解散，不再可访问/可聊天
- 自建群创建后默认进入 `active`。
- 只有群主可以解散群。
- 群解散后：
  - 群资料仍可作为审计记录保留（是否前台可见由后续方案决定）；
  - 成员关系失效；
  - 不能继续发送消息；
  - 原邀请/申请流程全部终止。

### 7.2 群角色与权限
- 群内基础角色固定为三类：
  - 群主（owner）
  - 管理员（admin）
  - 普通成员（member）
- 群主权限：
  - 编辑群核心资料
  - 发布/更新群公告
  - 任命/取消管理员
  - 审批入群申请
  - 邀请他人入群
  - 移除成员
  - 转让群主
  - 解散群
- 管理员权限：
  - 编辑群核心资料
  - 发布/更新群公告
  - 审批入群申请
  - 邀请他人入群
  - 移除普通成员
- 普通成员权限：
  - 查看群资料
  - 接受邀请 / 主动退群
  - 群内正常聊天
- 权限边界：
  - 管理员除“不能处置其他管理员和群主”之外，其余前台管理权限与群主一致。
  - 管理员不能任命/取消其他管理员，不能转让群主。
  - 管理员不能移除群主，不能移除其他管理员。
  - 普通成员不能修改群资料、不能审批、不能邀请（是否允许成员邀请后续再定，本期先不开放）。

### 7.3 角色转换规则
- 创建群时，创建者自动成为群主。
- 群主可将普通成员提升为管理员。
- 群主可撤销管理员身份，管理员被撤销后变为普通成员。
- 群主转让仅能转给当前群成员。
- 群主转让成功后，群内必须始终且仅存在 1 个群主。
- 群主转让后的原群主身份待定：
  - 已确认：自动降为普通成员。

### 7.4 群资料规则
- 群基础资料至少包括：
  - 群名称
  - 群头像
  - 群简介
  - 群公告
  - 群类型
  - 加入模式
- 群公告应展示最近一次发布时间/更新人（如技术实现成本可接受，建议纳入）。
- 群简介用于静态介绍，群公告用于高频通知，两者不可互相替代。

### 7.5 群类型规则
- 本期至少支持：
  - 官方话题群
  - 玩家自建群
- 类型主要用于：
  - 数据建模与前端展示差异
  - 后续权限/运营策略扩展
- 本期不要求一次性定义所有未来类型，但要求模型可扩展，新增类型不应推翻现有结构。

### 7.6 加入模式规则
- 私密群：
  - 非成员不可直接加入；
  - 默认只能通过邀请进入；
  - 不允许提交入群申请。
- 申请加入群：
  - 非成员可提交申请；
  - 群主或管理员审批通过后加入；
  - 审批拒绝后需给用户明确结果提示。
- 完全公开群：
  - 非成员可直接加入；
  - 不需要审批；
  - 若已有邀请流程，也不影响直接加入。

### 7.7 申请与审批规则
- 同一用户对同一群在“待审批”状态下只能存在一条有效申请。
- 已是成员的用户不能重复申请。
- 群主与管理员都可处理申请。
- 申请处理结果至少包括：通过、拒绝。
- 申请被处理后，申请状态需要可追踪，避免重复审批。

### 7.8 邀请规则
- 邀请人至少应为群主或管理员。
- 被邀请人若已是成员，不可重复邀请。
- 公开群和申请群也允许通过邀请入群，邀请是一条并行入群路径。
- 私密群主要依赖邀请入群。
- 邀请接受后用户加入群；邀请拒绝后不加入群。

### 7.9 退出、移除、解散规则
- 普通成员可主动退群。
- 管理员可移除普通成员。
- 群主可移除管理员和普通成员，但不能“移除自己”，如需离开必须先转让群主或解散群。
- 管理员可执行除“处置其他管理员/群主”以及“转让群主/解散群”外的其他管理操作。
- 当群只剩群主 1 人时：
  - 可继续保留群；
  - 也可主动解散群。

### 7.10 官方群补充规则
- 官方群属于群类型中的一种，前台视角下仍只展示普通群成员结构，不额外暴露“官方管理位”。
- 官方群内会存在一个官方账号，用于身份标识、内容发布或后续运营协作。
- 当前阶段该官方账号**不直接承担群管理职责**，不作为独立前台管理角色。
- 官方群在前台仍沿用群主 / 管理员 / 普通成员三层角色体系。
- 后续若增加后台运营系统，再由后台能力补充对官方群的额外管理与干预，本期不纳入前台权限设计。

## 8. 验收标准
- 用户可以创建自建群，并在创建时设置基础资料和加入模式。
- 创建成功后，创建者自动成为群主。
- 群主可以编辑群头像、群简介、群公告。
- 群主可以任命管理员、取消管理员、转让群主。
- 三类角色在前后端均有明确权限控制，不允许越权操作。
- 系统支持私密群、申请加入群、公开群三种加入模式。
- 申请加入群支持“提交申请 -> 管理员/群主审批 -> 加入/拒绝”闭环。
- 邀请流程支持“发起邀请 -> 接受/拒绝 -> 入群/不入群”闭环。
- 成员支持主动退群，管理角色支持按权限移除成员，群主支持解散群。
- 群解散后不可继续作为有效聊天群使用。
- 群解散后，成员不可查看该群历史消息。
- 群模型可区分官方话题群与玩家自建群，并支持后续扩展更多类型。

## 9. 风险与待确认项
- 管理员与群主“完全一致”的权限范围，在实现上需明确是否包含“解散群”“编辑所有群资料”等高风险操作；当前需求按“除处置管理员/群主外其余一致”执行。
- 群解散是物理删除、逻辑删除还是逻辑关闭后前台不可见，需在技术方案中明确。
- 官方账号在官方群中的具体产品表现形式（昵称、标识、消息样式）待后续单独细化。
- 群数量上限、成员人数上限、申请有效期、邀请有效期等运营参数，本期先不细化，留待技术方案阶段补充。
